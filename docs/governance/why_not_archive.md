# Why-not Archive (不採用の記録)

将来「なぜこうしなかったのか？」という疑問に答えるための、不採用案の墓場です。

## 1. データベースでの直接集計 (SQL/Isar Count)
- **なぜ検討したか:** 実装が簡単で高速に見えるため。
- **なぜ却下したか:** ルールが動的に変わるドメインにおいて、DBの固定カラムでの集計は過去のリプレイ整合性を破壊するため。
- **教訓:** 状態は常に「イベント履歴 + ルールエンジン」から動的に導出されなければならない。

## 2. UI側でのバリデーションロジック保持
- **なぜ検討したか:** 入力時に即座にエラーを出したいため。
- **なぜ却下したか:** 複数のUI（Scorer, Viewer）でロジックが重複し、エンジンの挙動と乖離するリスクがあるため。
- **教訓:** UIは「薄く」保ち、すべての正当性判断はドメイン層のエンジンに集約する。
