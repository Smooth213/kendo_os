# ADR 010: Documentation = Runtime Knowledge (運用知識のガバナンス化)

## Status
Accepted

## Context (背景と課題)
Kendo Syncは、体育館という過酷な環境で高齢者を含む多様なユーザーによって運用される。従来の「コードが本体、取説は付属品」というアプローチでは、システムの変更と現場の運用知識の間にズレ（Drift）が生じ、致命的な運用トラブルを招く。我々は、ドキュメントを「AIと共存可能な競技運営OSの知識基盤（Knowledge Governance）」としてシステム自体に統合する必要がある。

## Decision (決定事項)
システムにおいて「Documentation = Runtime Knowledge」と定義し、以下のアーキテクチャを採用する。

1. **なぜMarkdown管理なのか (Why Markdown?)**
   - CI/CDパイプラインによる静的解析（差分検知、リンク切れ検知）を可能にするため。バージョン管理システム上でコードの変更と同一のPull Requestで統治・監査できる唯一のフォーマットである。
2. **なぜPDFを真実にしないのか (Why PDF is not the Truth?)**
   - PDFはバイナリであり、差分比較や部分的な自動更新が不可能であるため。PDFはあくまで現場（体育館）へ持ち込むための「一時点の射影（Projection）」として自動生成されるべき成果物とする。
3. **なぜAI検索前提なのか (Why AI-Search native?)**
   - 今後、運用中のトラブル解決や手順の検索において、キーワード検索だけでは人間のコンテキスト（文脈）を拾いきれないため。将来的なRAG（検索拡張生成）や全文検索（Vector Search）に耐えうるよう、初めから意味的なチャンク（塊）やメタデータを意識した構造化ドキュメントを必須とする。

## Consequences (結果)
* **Positive**: 取説の陳腐化が物理的に不可能になる。AIがシステムの運用ルールを完全に理解し、アシスタントとして機能する基盤が整う。
* **Negative**: 画面UIの軽微な変更であっても、スクリーンショットの再取得とMarkdownの更新がCIによって強制されるため、開発者の負担（プロセス）は増加する。
