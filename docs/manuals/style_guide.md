# Kendo Sync 家電メーカー品質マニュアル スタイルガイド

本ドキュメントは、Kendo Syncの「知識基盤」を構築・保守するすべての開発者・テクニカルライターが遵守すべき絶対憲法である。「素人でも絶対に迷わない」品質を担保するため、以下の原則を厳守すること。

## 1. 真実の単一情報源 (Single Source of Truth)
本システムのドキュメントは、**「Markdownファイルを唯一の真実」** とします。
- **絶対原則**: アプリ内のヘルプやPDFは、すべてMarkdownから自動生成します。
- **禁止事項**: PDFを直接編集することは禁止です。
- **禁止事項**: 画面の文言だけを変更することも禁止です。

## 2. ディレクトリ完全固定 [Step 1-2]
マニュアルの構造は以下に固定し、勝手な階層の追加を禁止する。
```text
docs/manuals/
 ├─ viewer/          (観客・閲覧向け)
 ├─ operator/        (運営・記録向け)
 ├─ quickstart/      (現場用3分ガイド)
 ├─ faq/             (よくある質問)
 ├─ troubleshooting/ (トラブル・パニック時導線)
 ├─ recovery/        (緊急紙運用フォールバック)
 ├─ glossary/        (用語集 ※Phase 2で追加)
 ├─ templates/       (記述テンプレート)
 ├─ images/          (UI等の画像 ※assets/manual_images/ へのシンボリックリンクまたは参照)
 ├─ search_index/    (検索インデックス格納)
 └─ pdf/             (生成されたPDF出力先)

```

## 3. 命名規約固定 [Step 1-3]

ファイル名は恒久的な識別子（Manual ID）として機能するため、一意かつ不変でなければならない。

* **形式**: `[役割]_[画面名・機能名].md`
* **⭕ 良い例**: `viewer_match.md`, `operate_match.md`
* **❌ 禁止例**: `new_final_manual_latest.md`, `match_manual_v2.md` (バージョンや状態を含めてはならない)

## 4. 文体ルール固定 (Appliance Quality Writing) [Step 1-4, 2-3, 4-2, 4-3, 4-4]
「高齢の先生」から「初見の保護者」まで、誰が読んでも認知負荷ゼロで理解できる文体を強制する。
- **敬語統一**: 「です・ます」調で統一する。偉そうな断定表現は避ける。
- **1文40文字以内監査 (Step 4-2)**: 長文を禁止し、句点（。）までの1文の長さを40文字以内に収めること。
- **「何が起きるか」先書き (Step 4-3)**: 操作の意味や状態確認を先に書かず、結果や影響を先書きする。
  - ❌ 悪い例: 同期状態を確認します
  - ⭕ 良い例: 通信が切れていると、他の端末へ点数が反映されません。
- **操作結果明記 (Step 4-4)**: ボタンを押した直後に画面がどうなるか、必ず明記する。
  - ⭕ 良い例: 保存すると大会一覧へ戻ります。
- **専門用語禁止**: 開発用語は禁止用語辞書（`forbidden_words.md`）に従い、すべて平易な代替語に置き換える。

## 5. 視覚的ドキュメント化規則 (Visual Documentation System)

「読む」マニュアルではなく「見てわかる」マニュアルにするための厳格なキャプチャ・図解規約。

### 5-1. スクリーンショットポリシー (Step 3-1)
マニュアル内のすべての画面キャプチャは、以下の条件を完全に統一して撮影すること。
- **サイズと比率**: 統一デバイス（例: iPad Pro 11インチなどのタブレット比率）で撮影する。
- **テーマと環境**: ライトモード（Light Theme）に統一し、UI言語は日本語とする。

### 5-2. キャプチャ命名完全固定 (Step 3-2)
画像ファイル名は以下のフォーマットを遵守し、`assets/manual_images/` に配置すること。
- **形式**: `[Manual ID]_[連番]_[状態・操作].png`
- **⭕ 良い例**: `viewer_match_01_start.png`, `operate_match_timer.png`
- **❌ 悪い例**: `screen1.png`, `IMG_1234.jpg` (Manual IDがない、拡張子違い)

### 5-3. 番号付き図解ルール (Step 3-3)
画面全体の構成を説明する際は、画像内に丸数字（①、②、③...）を配置し、画像の下部にその番号と対応する要素名をリストアップすること。
- **記述例**:
  ① 大会名・試合名
  ② 残り時間（タイマー）
  ③ 現在の得点

### 5-4. 注釈テンプレート固定 (Step 3-4)
画像の特定箇所（タップするボタンやエラー表示など）を強調し操作を説明する場合、以下の注釈フォーマットを絶対ルールとする。
- **画像加工のスタイル**: 対象箇所を **「赤枠」** で囲み、そのすぐ横に **「丸数字（①など）」** を配置する。手書きの丸や矢印は使用禁止。
- **説明文のスタイル**: 画像の直下に `番号 + 1行説明` の形式で短く記述する。
- **記述例**:
  ① 【確定】ボタンを長押しします。

## 6. 検索・ハイライト対応要件 (Search & Highlight)

* **見出しID固定**: 主要な見出しには `{#id-name}` を付与し、リンク切れを防ぐ。
* **検索キーワード辞書**: `docs/manuals/search_dictionary.json` をメンテナンスし、表記ゆれを吸収する。

## 7. マニュアル導線固定ルール (Navigation Standard) [Phase 2]
ユーザーがどのページからアクセスしても絶対に迷子にならないよう、各ページの構造に以下の導線を必ず含めること。

### 7-1. パンくずリスト (Breadcrumbs)
ページの先頭には、必ずトップから現在のページまでの経路を配置する。
- **書式**: `総合ホーム &gt; 親カテゴリ &gt; ページ名`

### 7-2. フッター固定導線 (Footer Navigation)
ページの末尾には、以下の2つのセクションを必ず配置すること。
1. **関連ページ**: 業務上関連するページへのリンク。ViewerとOperatorの境界を越える場合は相互導線（Step 2-3）を設ける。
2. **困ったとき**: トラブル発生時の緊急導線を固定で配置する（Step 2-4）。

**▼ ページ末尾のテンプレート**
```markdown
---

## 関連ページ
- [関連ページA](link_a.md)
- [関連ページB](link_b.md)
- 👁️ [観客(または運営)画面の見え方はこちら](../viewer/xxx.md)

## 困ったとき
- 📡 [通信が切れた](../quickstart/index.md#network-error)
- 🥶 [画面が止まった](../quickstart/index.md#app-freeze)
- 📝 [点数を間違えた](../quickstart/index.md#undo-mistake)

```
